## C) Relationships and Cardinality

- users 1 -> many users/{userId}/progress
- users 1 -> many users/{userId}/daily_insights (rules path)
- categories 1 -> many lessons
- lessons 1 -> many lesson signs
- word_groups 1 -> many words
- wordGroups 1 -> many words (variant)

Logical links:

- progress document id == lessonId
- lesson.categoryId references categories document id
- signs.lessonId references lesson document id
- sign_practice_logs references learnerId, lessonId, signId
- issues may reference lessonId and signId

## D) Access Control Summary (from firestore rules)

- admins: owner/admin read, admin write
- users: owner/admin read+update+delete, authenticated create
- users/{userId}/progress: owner/admin read+write
- categories and nested lessons/signs: authenticated read, admin write
- signs (global): authenticated read, admin write
- word_groups and words: authenticated read, admin write
- issues: authenticated create, admin read/update/delete
- settings: authenticated read, admin write
- audit_logs: admin read/write
- sign_practice_logs: authenticated create, admin read
- daily_insights top-level: authenticated read, admin write

## E) Query and Index Notes

Observed query patterns that may require Firestore indexes:

- users ordered by createdAt desc
- users filtered by lastLoginAt and createdAt
- categories ordered by order
- lessons ordered by order under category
- signs ordered by order (global and nested)
- word_groups ordered by order
- words ordered by order
- issues filtered by status, priority, category + order by createdAt
- sign_practice_logs filtered by learnerId/lessonId/timestamp + order by timestamp
- daily_insights filtered by date and isActive, then by isActive + order by date desc
- progress filtered by categoryId and status in user scope

If Firestore prompts for composite indexes at runtime, generate them from the provided console links and store in firestore indexes configuration.

## F) Inconsistencies and Migration Risks

1. word_groups versus wordGroups

- Two collection names represent similar domain data.
- Risk: user pages and admin pages can diverge.

2. settings/maintenance versus system/maintenance

- Two paths are used for maintenance mode.
- Rules explicitly define settings, not system.

3. sign_images usage is mostly legacy

- Active Dart code favors URLs and storage references.
- Python utility still writes base64 images to sign_images.
